## Wnioski

Macierz: `baseline-py314` (przed refaktoryzacją), `refactor-py314-nocache` /
`refactor-py314` (po refaktoryzacji async, GIL), `refactor-py314t-nocache` /
`refactor-py314t` (free-threaded). Liczby poniżej to p50 / RPS.

### 1. Odblokowanie event loopu (baseline → refactor-py314-nocache)

- `mixed_80_20@c20`: **p50 233 ms → 6 ms (−97 %)** — odczyty przestały czekać
  za commitami gita wykonywanymi na event loopie. To był główny cel refaktoryzacji.
- Scenariusze czystego odczytu (`note_html`) i czystego zapisu (`seed_write`,
  `note_patch`) bez zmian — zgodnie z oczekiwaniami (nie były ofiarą blokady).
- `history@c50` wciąż fatalne (30 s, 18 timeoutów) — wąskim gardłem nie był
  event loop, tylko GIL: 10 równoległych przejść dulwicha po historii
  (czysty Python) wykonywało się sekwencyjnie.

### 2. Cache z epokami (nocache → cache, na obu runtime'ach)

- `history@c10`: 7 535 ms → **15,6 ms**; `history@c50`: 30 378 ms → **100 ms**,
  timeouty 18 → 0 (na 3.14); na 3.14t analogicznie 17 750 ms → 88 ms.
- Mikropomiar search (400 zapytań, 4 rotujące frazy, 500 notatek, 3.14t):
  p50 14,8 ms → **0,002 ms** na trafieniu; suma 5 850 ms → 59 ms.
- **Werdykt wg reguły ≥20 %: oba cache (search + history) zostają** — poprawa
  o 2–4 rzędy wielkości na scenariuszach docelowych.

### 3. Free-threading 3.14t (refactor-py314* → refactor-py314t*)

Zyski (równoległość realnie działa):
- `note_list@c50`: p50 1 704 ms → **131 ms**, RPS 27 → **278** (~10×) — SQL +
  stat() 500 plików w threadpoolu wreszcie się zrównolegla.
- `history@c50` bez cache: 30 378 ms → 17 750 ms i **0 timeoutów** (dulwich
  w wątkach liczy się równolegle, ~2×).
- `note_html@c50`: p50 112 → 63 ms, RPS +19 %.

Koszty (podatek free-threadingu + czysto-pythonowy fallback SQLAlchemy):
- SQLAlchemy cyextension nie deklaruje wsparcia ft → wymuszamy
  `DISABLE_SQLALCHEMY_CEXT_RUNTIME`; efekt: `search_inproc@t1` 5,4 → 14,5 ms
  (~2,7×), `reindex_total` 532 → 786 ms (+48 %).
- Pojedyncze przejście dulwicha: 688 → 735 ms (+7 %) — podatek interpretera.
- **RSS: 168 → 352 MB (~2×)** — narzut buildu free-threaded.

### 4. Rekomendacja

Refaktoryzacja async + cache dają zdecydowaną większość zysku i są bezpłatne.
3.14t opłaca się przy obciążeniu współbieżnym wieloma odczytami/listowaniami
(`note_list@c50` ×10), kosztem ~2× RAM i wolniejszych operacji jednowątkowych
opartych o SQLAlchemy (reindex, pojedynczy search). Gdy SQLAlchemy wyda
cyextension z deklaracją free-threading (lub po przejściu na nowszą wersję),
koszt #3 w dużej mierze zniknie — warto wtedy powtórzyć macierz.
